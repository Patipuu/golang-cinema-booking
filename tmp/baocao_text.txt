1.1. L ịch sử hình thành và đặc điểm ngôn ngữ  
1.1.1. B ối cảnh ra đ ời 
Ngôn ngữ lập trình Go (tên chính thức) hay Golang (tên thông dụng để phân biệt khi 
tìm kiếm) được khởi xướng nội bộ tại Google vào năm 2007 bởi ba kỹ sư: Robert 
Griesemer, Rob Pike và Ken Thompson -  trong đó Ken Thompson là đồng tác giả của 
ngôn ngữ C và hệ điều hành Unix. Go được công bố mã nguồn mở vào năm 2009 và 
phát hành phiên bản ổn định đầu tiên (Go 1.0) vào tháng 3 năm 2012. 
Sự ra đời của Go không xuất phát từ mục tiêu học thuật thuần túy, mà từ một vấn đề 
kỹ thuật thực tế: cơ sở hạ tầng phần mềm tạ i Google vào cuối những năm 2000 đang 
phải vận hành các hệ thống quy mô cực lớn với hàng triệu dòng code C++ và Java. Ba 
điểm đau chính được xác định là:  
• Thời gian biên dịch quá chậm với C++, làm giảm năng suất lập trình viên.  
• Mô hình concurrency phức tạp và dễ lỗi với Thread/Mutex truyền thống trong 
Java và C++. 
• Thiếu một ngôn ngữ hệ thống hiện đại kết hợp được tốc độ của C với sự an toàn và năng suất của các ngôn ngữ hiện đại hơn.  
Go được thiết kế để giải quyết đồng thời cả ba vấn đề trên, với triết lý th iết kế cốt lõi: 
"Make it simple, but not simpler.”  
1.1.2. Đ ặc điểm ngôn ngữ  học cơ b ản 
Từ góc độ phân loại ngôn ngữ lập trình, Go sở hữu một tập hợp đặc điểm khá độc đáo, 
không hoàn toàn thuộc về một trường phái nào:

 a) Kiểu tĩnh biên dịch tĩnh (Statically Typed, Compiled) 
Go là ngôn ngữ kiểu tĩnh (statically typed ): kiểu dữ liệu của mọi biến phải được xác 
định tại thời điểm biên dịch (compile time ), không phải lúc chạy (runtime). Điều này 
khác biệt cơ bản với Python hay JavaScript (kiểu động -  dynamically typed ). 
Tuy nhiên, Go bổ sung cơ chế type inference thông qua toán tử :=, cho phép trình biên 
dịch tự suy luận kiểu từ giá trị khởi tạo, giúp code ngắn gọn hơn mà không mất đi sự

khai (deployment ) đáng kể: chỉ cần copy một file binary là có thể chạy trên bất kỳ máy

 1.2. S ức mạnh của Go trong xử  lý đồng thời (Concurrency)  
1.2.1. Lý thuy ết - Mô hình Concurrency c ủa Go  
a) Phân biệt Concurrency và Parallelism  
Trước khi phân tích cơ chế của Go, cần làm rõ hai khái niệm thường bị nhầm lẫn 
trong tài liệu kỹ thuật:  
• Concurrency  (đồng thời): Khả năng cấu trúc  chương trình để xử lý nhiều tác 
vụ trong một khoảng thời gian, nhưng không nhất thiết tại cùng một thời điểm 
vật lý. Đây là tính chất của thiết kế chương trình . 
• Parallelism  (song song): Các tác vụ thực sự chạy đồng thời  trên nhiều CPU 
core tại cùng một thời điểm vật lý. Đây là tính chất của phần cứng và 
scheduling.  
Rob Pike - một trong ba tác giả của Go -  phát biểu: "Concurrency is about dealing 
with lots of things at once. Parallelism is about doing lots of things at once." Go được 
thiết kế để hỗ trợ concurrency một cách tự nhiên ở cấp độ ngôn ngữ, và khi có nhiều 
CPU core, Go runtime tự động tận dụng parallelism.  
b) Goroutine - Luồng xanh (Green Thread) của Go  
Goroutine là đơn vị thực thi đồng thời của Go. Về bản chất, gorou tine là green thread  
- luồng được quản lý bởi Go runtime, không phải bởi hệ điều hành (OS thread). 
Cơ chế hoạt động bên trong:  
Go runtime sử dụng mô hình M:N scheduling: M goroutine được ánh xạ lên N OS 
thread, trong đó N thường bằng số CPU core (kiểm soát bởi GOMAXPROCS). Bộ lập 
lịch (scheduler ) của Go là work -stealing scheduler -  các OS thread rảnh rỗi có thể 
"đánh cắp" goroutine từ hàng đợi của các thread khác để tối ưu hóa sử dụng CPU. 
Goroutine được khởi tạo với stack cực nhỏ (mặc định 2KB trong Go  1.4 trở đi, so với 
8MB mặc định của OS thread trên Linux), và stack này có thể tự mở rộng (growable 
11 
 stack ) khi cần thiết, tối đa lên đến 1GB (có thể cấu hình). Đây là lý do một chương 
trình Go có thể khởi tạo hàng triệu goroutine mà không cạn kiệt bộ nhớ.

c) Channel -  Cơ chế giao tiếp giữa các Goroutine  
Go theo triết lý thiết kế CSP (Communicating Sequential Processes ) - một mô hình 
toán học được Tony Hoare đề xuất năm 1978. Nguyên tắc cốt lõi của CSP trong Go 
được đúc kết thành một câu:  
"Do n ot communicate by sharing memory; instead, share memory by communicating."  
Channel  là cơ chế cụ thể hóa nguyên tắc này: thay vì nhiều goroutine cùng truy cập 
một vùng nhớ chung (và cần mutex để đồng bộ), các goroutine giao tiếp bằng cách 
gửi/nhận dữ liệu q ua channel. 
Channel có hai dạng:  
• Unbuffered channel  (make(chan T)): người gửi bị block cho đến khi người 
nhận sẵn sàng - đồng bộ hóa tự nhiên. 
• Buffered channel (make(chan T, n)): người gửi chỉ bị block khi buffer đầy -  
tách rời nhà sản xuất và người tiêu thụ.

 1.3.2. So sánh vớ i các l ựa chọn thay th ế 
Để khẳng định tính phù hợp của Go, cần đối chiếu với các ngôn ngữ thay thế khả thi:  
Go so với Node.js (JavaScript): Node.js sử dụng mô hình event loop đơn luồng với 
callback/Promise, phù hợp cho I/O -bound workload đơn giản. Tuy nhiên, khi cần xử 
lý logic tính  toán phức tạp (tính giá vé với nhiều biến động, kiểm tra conflict lịch 
chiếu), Node.js bị hạn chế bởi single -threaded nature. Go xử lý cả I/O -bound lẫn 
CPU -bound đồng thời hiệu quả hơn nhờ M:N scheduler. 
Go so với Java (Spring Boot): Java có hệ sinh thái mature và mạnh, nhưng chi phí 
khởi động JVM (~1 –3 giây) và bộ nhớ baseline cao (~200 –500MB) là bất lợi trong 
môi trường microservices và containerized deployment. Go khởi động trong mili giây 
và tiêu thụ RAM thấp hơn một bậc. 
Go so với Python (FastAPI): Python với GIL ( Global Interpreter Lock ) không thể 
thực sự tận dụng multi -core CPU cho CPU -bound tasks. FastAPI dùng asyncio nhưng 
vẫn bị giới hạn bởi GIL. Go không có GIL -  mọi goroutine đều có thể chạy song song 
thực sự trên nhiều core.

truy cập đồng thời không an toàn vào vùng nhớ chung - một lỗi cực kỳ khó debug 
trong các ngôn ngữ như C++ hay Java.

4.1. Middleware là gì? Phân tích từ  góc độ  ngôn ngữ  học 
4.1.1. Đ ịnh nghĩa và cơ ch ế hoạt động 
Trong Go, middleware không phải là một từ khóa hay cấu trúc ngôn ngữ riêng biệt -  
nó là một design pattern được xây dựng hoàn toàn từ hệ thống kiểu (type system ) của 
ngôn ngữ. Cụ thể, middleware trong Go là một hàm có chữ ký (signature) dạng:  
func(next http.Handler) http.Handler  
Để hiểu tại sao chữ ký này lại là nền tảng, cần phân tích  từng thành phần:  
• http.Handler là một interface được định nghĩa trong thư viện chuẩn của Go: bất 
kỳ kiểu nào có phương thức ServeHTTP(ResponseWriter, *Request) đều thỏa 
mãn interface này -  đây là cơ chế structural typing đặc trưng của Go.  
• http.HandlerFunc là một function type được khai báo là type HandlerFunc 
func(ResponseWriter, *Request), và nó cài đặt ServeHTTP bằng cách tự gọi chính nó -  cho phép chuyển đổi linh hoạt giữa hàm thông thường và interface. 
• Tham số next http.Handler biểu diễn handler tiếp th eo trong chuỗi xử lý -  đây 
là cách Go hiện thực hóa mô hình xử lý tuần tự mà không cần framework.  
Middleware hoạt động theo nguyên lý bọc ngoài (wrapping): mỗi middleware nhận một handler, tạo ra một handler mới bao quanh handler gốc, và trả về handler mới  đó. 
Request đi vào từ middleware ngoài cùng, xuyên qua từng lớp, đến handler cuối, rồi response đi ngược ra. Đây là cấu trúc Decorator Pattern được thể hiện thuần túy qua 
hệ thống kiểu của Go mà không cần class hay kế thừa.

5.1 Xử  Lý Đ ồng Th ời - Bài Toán Đ ặt Gh ế Trùng Nhau (Double Booking)  
5.1.1 B ản Ch ất Của Vấn Đề 
Khi nhiều người dùng cùng thao tác trên một dữ liệu tại cùng một thời điểm, hệ thống 
dễ rơi vào tình trạng race condition -  tức là kết quả cuối cùng phụ thuộc vào thứ tự 
thực thi ngẫu nhiên của các luồng xử lý, chứ không phải theo logic nghiệp vụ mà ta mong muốn.  
Trong bài toán đặt ghế xem phim, tình huống điển hình diễn ra như sau: Goroutine A 
(User 1) và Goroutine B (User 2) đều đọc trạng thái ghế G5 và thấy nó đang còn 
trống. Cả hai đều vượt qua bước kiểm tra và tiến hành ghi "đã đặt" vào cơ sở dữ liệu. 
Kết quả là ghế G5 bị bán cho cả hai người -  đây chính là Double Booking.  
Điểm mấu chốt cần hiểu là: kiểm tra ở tầng ứng dụng (application layer) là không đủ. 
Dù code Go có kiểm tra if seat.Status == "available" trước khi ghi, nhưng nếu hai 
goroutine cùng đọc dữ liệu trước khi một trong hai ghi xuống, điều kiện kiểm tra đó 
trở nên vô nghĩa. Vấn đề này càng trầm trọng hơn khi hệ thống chạy nhiều instance 
song song (horizontal scaling), vì lúc đó nhiều process hoàn toàn độc lập cùng truy 
cập một cơ sở dữ liệu.  
Giải pháp đúng đắn là đẩy cơ chế kiểm soát xuống tầng cơ sở dữ liệu, nơi PostgreSQL 
đảm bảo tính ACID cho mọi transaction:  
Tính Atomicity (nguyên tử) đảm bảo toàn bộ transaction hoặc thành công hoàn toàn, 
hoặc được hoàn tác hoàn toàn -  không tồn tại trạng  thái nửa vời. Tính Consistency 
(nhất quán) đảm bảo dữ liệu luôn hợp lệ theo các ràng buộc đã định trước và sau 
transaction. Tính Isolation (cô lập) đảm bảo các transaction đang chạy song song 
không nhìn thấy dữ liệu trung gian của nhau. Tính Durability (b ền vững) đảm bảo dữ 
liệu đã được commit sẽ tồn tại vĩnh viễn kể cả khi hệ thống gặp sự cố.

 5.1.2 Cơ Ch ế SELECT ... FOR UPDATE Ho ạt Động Như Th ế Nào ?  
Câu lệnh SELECT ... FOR UPDATE là một dạng pessimistic locking  (khóa bi quan) 
- tức là ta giả định rằng xung đột sẽ xảy ra , nên chủ động khóa tài nguyên trước khi xử 
lý. Khi một transaction thực thi câu lệnh này, PostgreSQL đặt một row -level lock lên 
hàng đó. Mọi transaction khác muốn đọc hàng đó với FOR UPDATE sẽ phải chờ  cho 
đến khi transaction đầu tiên kết thúc (commit hoặc rollback). Nhờ vậy, chỉ một 
transaction tại một thời điểm có thể kiểm tra và sửa trạng thái của một ghế.

7.1.1. Postman Là Gì  ? 
Postman là một ứng dụng desktop/web miễn phí được lập trình viên toàn thế giới sử 
dụng để kiểm thử API mà không cần viết code frontend. Thay vì phải mở trình duyệt 
hay xây dựng giao diện người dùng, Postman cho phép bạn gửi các HTTP request (GET, POST, PUT, DELETE...) trực tiếp đến server, xem kết quả trả về theo thời gian thực và kiểm tra API có hoạt động đúng như kỳ vọng hay không.  
Dành cho người mới học:  Hãy tưởng tượng bạn đang xây một nhà hàng (backend 
server). Postman chính là người khách hàng giả lập - họ vào quán, gọi món (gửi 
request), và bạn kiểm tra xem bếp có nấu đúng món không (kiểm tra response). Bạn không cần xây cả mặt tiền nhà hàng (frontend) chỉ để test xem bếp có nấu được không.

Swagger  (còn gọi là OpenAPI Specification) là một tiêu chuẩn quốc tế để mô tả và tài

Lợi ích cụ thể với dự án:  
• Frontend developer biết chính xác c ần gửi gì, nh ận lại gì từ mỗi endpoint. 
• Thầy cô khi ch ấm điểm có th ể test API trự c tiếp trên trình duy ệt, không c ần cài 
Postman hay dùng terminal. 
• Toàn nhóm  có tài li ệu thống nh ất, tránh hi ểu nhầ m về cấu trúc d ữ liệu.

8.1. Xác Th ực Và Phân Quyề n Ngư ời Dùng (Jwt + Middleware Pattern) 
8.1.1 Lý thuy ết JSON Web Token (JWT ) 
JWT (JSON Web Token) là m ột tiêu chu ẩn mở (RFC 7519) đ ịnh nghĩa cách truy ền 
thông tin an toàn gi ữa các bên dư ới dạng đố i tượng JSON. Token g ồm 3 ph ần: 
header.payload.signature  
• Header: Khai báo thu ật toán mã hóa (HS256 –  HMAC v ới SHA -256).  
• Payload (Claims): D ữ liệu cần truy ền (user_id, email, th ời gian h ết hạn). 
• Signature: HMAC -SHA256(base64url(header) + "." + base64url(payload), 
secret). Ch ỉ server bi ết secret, đ ảm bảo token không bị  giả mạo. 
JWT là  Stateless: Server không c ần lưu session, ch ỉ cần verify ch ữ ký là biế t ngay ai 
đang g ọi. Điều này giúp hệ  thống scale theo chi ều ngang (horizontal scaling) không 
cần shared session store. 
8.1.2 Lý thuy ết Middleware Pattern trong Go  
Go không có class hay k ế thừa. Thay vào đó, Go s ử dụng Higher- Order 
Functions  (hàm nh ận hàm làm tham s ố, trả về hàm). Middleware trong Go là m ột hàm 
nhận http.Handler và trả về http.Handler khác đã đư ợc "bọc" thêm l ogic. 
Đây là ứ ng dụ ng củ a Decorator Pattern  – một trong các Design Pattern (GoF) nhưng 
được diễn đạt cực kỳ ngắn gọn và t ự nhiên nh ờ tính năng "functions as first -class 
citizens" c ủa Go.  
Request → [AuthMiddleware] → [LoggerMiddleware] → [Handler] 
Mỗi Middleware trong chu ỗi chỉ làm m ột việc, tuân theo  Single Responsibility 
Principle (SRP) trong SOLID.  
 
 
 
8.1.3 Lý thuy ết Context (Truy ền dữ liệu xuyên su ốt request) 
Context.Context  trong Go là m ột công c ụ cơ bản để:

 8.2. Xử Lý Đ ồng Thời Và Race Condition (Concurrency Control)  
8.2.1 Lý thuy ết Concurrency trong Go: Goroutines và Scheduler  
Go cung c ấp Goroutines  – lightweight threads đư ợc quản lý b ởi Go Runtime, không 
phải OS thread. M ỗi goroutine ban đ ầu chỉ chiếm ~2KB stack (so v ới 1-8MB c ủa OS 
thread). 
Go Runtime s ử dụng mô hình M:N threading  (M goroutines ánh x ạ vào N OS threads) 
thông qua b ộ scheduler G -P-M: 
• G (Goroutine): Đơn v ị thực thi.  
• P (Processor): Ng ữ cảnh th ực thi, m ỗi P có local run queue. 
• M (Machine/OS thread): Thread th ực sự. 
Điều này cho phép hàng nghìn goroutines t ồn tại song song mà không tố n nhi ều bộ 
nhớ, phù h ợp cho h ệ thống xử  lý đồng th ời nhiều yêu c ầu đặt vé. 
 
8.2.2 Lý thuy ết CSP (Communicating Sequential Processes) và Channels  
Go áp d ụng tri ết lý CSP của Tony Hoare:  "Do not communicate by sharing memory; 
instead, share memory by communicating."  Thay vì các goroutine cùng truy c ập một 
biến shared và lock nó, chúng giao ti ếp qua  Channels  – các ố ng dẫn an toàn type -safe. 
Channel trong Go hoạ t động theo cơ ch ế FIFO (First In, First Out) và là concurrent -
safe theo thi ết kế của ngôn ng ữ (không cầ n lock th ủ công).  
• Unbuffered channel: Sender block cho đ ến khi Receiver s ẵn sàng và ngượ c lại 
– đảm bảo đồng bộ  hóa tuy ệt đối giữa hai goroutine.  
• Buffered channel: Sender ch ỉ block khi buffer đ ầy – cho phép phi đ ồng bộ có 
giới hạn. 
 
 
 
8.2.3 Lý thuy ết sync.Mutex và sync.RWMutex  
Khi c ần chia s ẻ state (ví d ụ: map các providers), Go cung c ấp sync.Mutex  (khóa đ ộc 
quyền) và  sync.RWMutex  (cho phép nhi ều reader đ ồng th ời, một writer đ ộc quy ền). 
128 
 sync.RWMutex  phù hợ p khi đọ c nhi ều hơn ghi – giúp t ối ưu throughput so v ới Mutex 
thông thư ờng. 
 
8.2.4 Lý thuy ết Distributed Lock v ới Redis (SetNX) 
Trong môi trư ờng phân tán (nhi ều instance server), sync.Mutex  chỉ hoạt động trong 
cùng m ột process. Đ ể đồng bộ  xuyên các server, c ần Distributed Loc k. 
Redis cung c ấp lệnh SETNX  (SET if Not eXists): Atomically gán giá trị  cho key n ếu 
chưa t ồn tại. Tính atomic đư ợc đảm bảo bởi Redis single -threaded model –  không th ể 
có hai client nào cùng  SETNX  thành công vớ i cùng key.  
Kết hợp SETNX  với EXPIRE  (hoặc SET key value NX EX seconds), ta t ạo được 
khóa phân tán có TTL (Time -To-Live), tự  giải phóng n ếu client crash. 
 
8.2.5 Lý thuy ết Pessimistic Locking trong Database  
SELECT ... FOR UPDATE  trong PostgreSQL th ực hiện Row -Level Lock  – khóa 
cứng các hàng đã ch ọn cho  đến khi transaction k ết thúc. Các transaction khác c ố đọc 
các hàng này v ới FOR UPDATE  sẽ bị block hoặ c fail ngay (v ới NOWAIT).  
Đây là k ỹ thuật Pessimistic Locking  – phù hợ p cho d ữ liệu tranh ch ấp cao như ghế  
xem phim, vì gi ả định xung đ ột sẽ xảy ra và ngă n chặn từ đầu.

8.3. Hệ  Thống Real -Time Và Broadcast Tr ạng Thái Ghế  (Websocket)  
8.3.1 Lý thuy ết WebSocket Protocol  
HTTP là giao th ức request -response: Client h ỏi → Server trả  lời. Với yêu c ầu "cập 
nhật trạng thái gh ế theo th ời gian th ực cho t ất cả người đang xem cùng su ất chiếu", 
HTTP không đ ủ – Server không th ể chủ động push d ữ liệu. 
WebSocket  (RFC 6455) giả i quy ết vấn đề này: B ắt đầu bằng HTTP Upgrade 
handshake, sau đó duy trì  kết nối full -duplex  liên t ục. Cả Client và Server đ ều có th ể 
gửi bất kỳ lúc nào.

 8.3.2 Lý thuy ết Fan -Out Pattern  
Khi m ột sự kiện xảy ra (gh ế A5 b ị lock), c ần thông báo t ới N clients đang xem cùng 
showtime. Đây là  Fan-Out pattern: M ột producer, nhiề u consumer.  
Go's Hub pattern tri ển khai Fan- Out qua:  
• Single producer: Hàm  BroadcastSeatUpdate  gửi một message 
vào hub.broadcast. 
• Single consumer goroutine  (Hub.Run): Nh ận từ channel và g ửi đến tất 
cả client.Send  channe ls. 
• N goroutines  (WritePumps): M ỗi goroutine đọ c từ client.Send  và gửi qua 
WebSocket connection. 
Kiến trúc này tránh đư ợc việc phải lock khi iterate qua map clients, vì ch ỉ có m ột 
goroutine duy nhấ t làm vi ệc đó.  
 
8.3.3 Lý thuy ết Goroutine per Connection  
Go Runtime c ực kỳ hiệu quả với goroutine: M ột server có th ể xử lý hàng chụ c nghìn 
WebSocket connections đ ồng thờ i, mỗi connection có 2 goroutines (ReadPump + 
WritePump), tương đương ~80K goroutines v ới tổng bộ  nhớ khoảng 160MB, điề u 
không thể  với OS threads.

 8.4. Ki ến Trúc Ph ần Mềm (Clean Architecture & Depend

8.4.1 Lý thuyế t Clean Architecture  
Clean Architecture (Robert C. Martin) đ ịnh nghĩa các vòng tròn ph ụ thuộc: Domain 
→ Repository (Interface) → Service → Handler. M ỗi lớp chỉ phụ thuộc vào l ớp bên

Điều quan trọ ng nh ất: Business logic (Service layer) hoàn toàn không bi ết về 
Framework, Database, hay HTTP. Đi ều này cho phép thay thế  PostgreSQL b ằng 
MongoDB, thay th ế chi-router b ằng gin, mà không c ần sửa code logic.  
 
8.4.2 Lý thuy ết Interface trong Go (Implicit Implementation)  
 
Go không có t ừ khóa implements. M ột type t ự động th ỏa mãn m ột interface n ếu nó cài 
đặt tất cả các method c ủa interface đó. Đây là  Duck Typing có ki ểm tra t ại compile 
time –  linh hoạ t hơn Java/C# nhưng v ẫn type -safe. 
Interfaces trong Go thư ờng nhỏ và tập trung (Interface Segregation

9.4.1. Khó Khăn 1 - Kết Nối Go V ới PostgreSQL  
Vấn đề gặp phải:  Ban đầu nhóm gặp lỗi pq: SSL is not enabled on the server  khi kết

không đóng lại.

• PostgreSQL local m ặc định không b ật SSL, nhưng driver Go l ại yêu c ầu SSL 
theo m ặc định. 
• Không cấ u hình connection pool, d ẫn đến số lượng connection tăng không 
kiểm soát. 
Cách khắc phục:  
// Thêm ?sslmode=disable vào connection string

 // Cấu hình connection pool  
sqlDB , _ := db.DB()  
sqlDB .SetMaxOpenConns(25)        // T ối đa 25 connection m ở đồng th ời

 9.4.2. Khó Khăn 2 - Xử Lý Transaction Đ ặt Vé (Race Condition)  
 
Đây là v ấn đề phức tạp nhất của dự án. Khi 2 ngư ời dùng cùng lúc ch ọn cùng m ột ghế 
và nh ấn "Đ ặt vé" trong cùng m ột mili -giây, h ệ thống không đư ợc để cả hai đ ặt thành 
công. Tuy nhiên, n ếu xử lý khô ng khéo, có th ể xảy ra tình trạ ng **m ột ngư ời bị mất 
tiền nhưng không có vé** (đã thanh toán nhưng booking bị  hủy sau đó).  
 
Minh h ọa vấn đề (nếu không có transaction):  
Thời điểm T1 : User A ki ểm tra gh ế 12 → kết quả: "available"  
Thời điểm T1 : User B ki ểm tra gh ế 12 → kết quả: "available"   (đọc cùng lúc ) 
Thời điểm T2 : User A t ạo booking cho gh ế 12 → thành công  
Thời điểm T2 : User B t ạo booking cho gh ế 12 → cũng thành công ( ?!) 
→ K ết quả: 2 người cùng gi ữ 1 ghế - dữ liệu bị sai

10.2.1. Chuy ển Đổi Từ Monolithic Sang Microservices  
 
Kiến trúc hi ện tại - Monolithic (Nguyên kh ối): 
Toàn b ộ dự án hi ện tại là m ột ứng dụ ng Go duy nh ất chạy trên m ột server, x ử lý mọi 
thứ: xác th ực ngư ời dùng, qu ản lý phim, đ ặt vé, thanh toán... Đây g ọi là ki ến trúc 
Monolithic  (một khối). 
 
Monolithic phù h ợp cho giai đo ạn đầu vì đơn gi ản, dễ phát tri ển, dễ debug. Tuy nhiên 
khi hệ  thống lớ n lên, s ẽ phát sinh v ấn đề: 
- Một lỗi nhỏ trong module Thanh toán có th ể làm s ập toàn bộ  hệ thống kể cả module 
Xem phim.  
- Không thể  mở rộng từng ph ần độc lập: vào d ịp lễ, module Đ ặt vé c ần nhi ều server 
hơn, nhưng vì t ất cả trong 1 ứng dụ ng, ph ải nhân rộ ng cả  hệ thống - tốn kém và lãng 
phí. 
- Khi team lớ n hơn (10+ ngư ời), nhi ều ngư ời cùng s ửa một codebase gây ra conflict 
liên t ục. 
 
Kiến trúc tương lai -  Microservices (Đa d ịch vụ): 
Tách ứ ng dụ ng th ành các **service đ ộc lập**, m ỗi service ch ịu trách nhi ệm một 
nghiệp vụ cụ thể, có database riêng, deploy riêng:  
Lợi ích cụ thể:  
• Fault Isolation (Cô l ập lỗi): Nếu Payment Service b ị lỗi, ngư ời dùng v ẫn xem 
được danh sách phim và su ất chiếu bình thư ờng - chỉ chức năng thanh toán b ị 
ảnh hư ởng. 
• Independent Scaling (M ở rộng độc lập): Dịp Tết khi lư ợng đặt vé tăng đ ột biến, 
chỉ cần nhân s ố instance c ủa Booking Servi ce lên -  không c ần đụng đế n User

10.2.2. Ứng Dụ ng Redis Đ ể Caching  
Redis là gì?  Redis (Remote Dictionary Server) là một cơ sở dữ liệu in -memory  - nghĩa 
là nó lưu dữ liệu trực tiếp trên RAM thay vì ổ cứng. Điều này khiến tốc độ đọc/ghi của 
Redis nhanh hơn PostgreSQL khoảng 100-1000 lần . Redis thường được dùng để 
caching  - lưu tạm kết quả của các truy vấn thường xuyên và tốn tài nguyên.  
Vấn đề hiện tạ i cần caching giải quyết:  
Mỗi khi người dùng mở trang chủ xem danh sách phim, hệ thống phải:  
1. Kết nối đến PostgreSQL. 
2. Thực thi câu query SELECT * FROM movies WHERE status = 'showing' 
ORDER BY created_at DESC . 
3. PostgreSQL đ ọc từ ổ cứng, trả  về kết quả. 
4. Go serialize k ết quả thành JSON.  
5. Trả về cho ngư ời dùng.  
Nếu có 1,000 người cùng lúc mở trang chủ → 1,000 lần query xuống PostgreSQL → 
database bị quá tải dù dữ liệu danh sách phim gần như không thay đổi trong 1- 2 giờ .

func (s *MovieService ) GetNowShowingMovies() ([]Movie , error)  { 
    cacheKey :=  "movies:now_showing"  
     
    // Bư ớc 1: Kiể m tra cache trư ớc 
    cached , err :=  s.redis .Get(context .Background(), cacheKey ).Result() 
    if err ==  nil { 
        // Cache HIT -  có dữ liệu trong Redis, trả  về ngay  
        // Không c ần đụng đến PostgreSQL  
        var movies []Movie 
        json.Unmarshal([]byte(cached ), &movies ) 
        return  movies , nil 
    } 
     
    // Bư ớc 2: Cache MISS -  không có trong Redis, t ruy vấ n PostgreSQL  
    movies , err :=  s.movieRepo .FindNowShowing()  
    if err !=  nil { 
        return  nil, err 
    } 
     
    // Bư ớc 3: Lưu k ết quả vào Redis đ ể những lầ n sau dùng l ại 
    // TTL = 30 phút - sau 30 phút Redis t ự xóa, l ần sau s ẽ query DB l ại để lấy dữ liệu 
mới 
    data , _ := json.Marshal(movies ) 
    s.redis .Set(context .Background(), cacheKey , data, 30*time.Minute)  
     
    return  movies , nil 
}
